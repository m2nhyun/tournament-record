# Keep Rules (Non-Negotiables)

이 문서는 지금까지 작업에서 반드시 유지해야 하는 기준을 고정한다.

## 1) Auth / 계정

- 인증 책임은 `src/features/auth/services/auth.ts`에 유지한다.
- `getCurrentUser()`의 `Auth session missing` 예외 무시 처리 정책을 유지한다.
- 카카오 로그인 scope는 `profile_nickname profile_image`를 기본으로 유지한다.
- 카카오 사용자 로그아웃 시 Supabase sign-out 뒤 카카오 logout URL 연동을 유지한다.

## 2) DB 반영 방식

- 코드 변경과 DB 변경은 분리 관리한다. DB가 필요한 기능은 SQL 반영 전까지 미완료로 본다.
- 현재 운영 정책상 DB 마이그레이션은 SQL Editor 수동 실행을 기본으로 유지한다.
- 신규 SQL은 `supabase/migrations/*.sql`에 항상 남긴다.

## 3) 클럽 권한/설정

- 클럽 이름 변경은 owner만 가능해야 한다.
- 멤버에게 클럽명 편집 UI를 노출하지 않는다.
- 내 닉네임/개인 공개 설정은 멤버 목록의 `내 항목` 편집 모달에서만 변경한다.
- 클럽명 정규화 unique, 클럽 내 닉네임 정규화 unique 제약을 유지한다.

## 4) 경기 기록 UX

- 경기 기록 입력은 타이핑보다 선택형 UX를 우선한다.
- 팀 표기는 `사이드 1/2` 대신 선수 이름 기반으로 유지한다.
- 용어는 `세트/라운드` 혼용을 피하고 `게임` 중심으로 유지한다.
- 목표 게임(6/4) 선택 로직과 유효성 검증(6-4/7-6, 4-2/5-4)을 유지한다.
- 예외 기록(예: 3게임 종료)은 `진행/중단`으로 명시해 숨기지 않는다.

## 5) 상세/히스토리 표시

- 경기 상세 상단은 `팀명 + 전체 스코어` 중심 배치를 유지한다.
- 게임별 상세 테이블은 선수명 헤더를 유지한다.
- 포인트는 미입력 시 노출하지 않고, 입력 시 게임 점수에 포함해 표시한다.

## 6) 상태 의미

- 현재 상태 라벨에서 `제출됨`은 사용하지 않는다.
- 사용자 관점 상태는 `기록됨` 중심으로 노출한다.
- 기본 경기 생성 상태는 `confirmed`로 유지한다.

## 7) UI 구현 원칙

- 인라인 편집 폼으로 화면을 복잡하게 만들지 않는다.
- 편집은 연필 아이콘 트리거 + 모달/다이얼로그 패턴을 기본으로 한다.
- 공통 모달은 `shadcn Dialog` 기반 컴포넌트(`src/components/ui/dialog.tsx`, `src/components/common/modal.tsx`)를 재사용한다.
